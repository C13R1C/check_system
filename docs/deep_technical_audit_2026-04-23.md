Date: 2026-04-23

## 1) Technical terms extracted from the codebase

### A. Backend architecture

- **Application Factory (`create_app`)**
  - Simple: the app is built by a function each time it starts.
  - Technical: Flask app initialization uses factory pattern, loading config, extensions, blueprints, hooks (`before_request`, `after_request`, `teardown_appcontext`).

- **Blueprints**
  - Simple: modules that split routes by feature.
  - Technical: modular routing with per-domain controllers (`reservations`, `notifications`, `inventory`, etc.) registered in app bootstrap.

- **Controller layer**
  - Simple: endpoints that receive user requests.
  - Technical: Flask route handlers orchestrating request parsing, authorization checks, service invocations, DB transactions, and response generation.

- **Service layer**
  - Simple: business logic separate from web pages.
  - Technical: domain services encapsulate workflows (reservations, debt, notification, cleanup), reducing controller complexity.

- **Repository pattern (partial)**
  - Simple: specialized data-access helpers.
  - Technical: repository modules exist for some domains (reservation/debt/inventory/lostfound), but data access is also done directly via ORM queries in controllers/services.

- **Context processors**
  - Simple: global data injected into templates.
  - Technical: app-level context processors inject notifications and CSRF token into Jinja render context.

- **Jinja filters**
  - Simple: helper functions for formatting output.
  - Technical: custom filters (`smart_title`, `normalize_spaces`, `role_label`, `status_label`) registered in Jinja environment.

- **Scoped session lifecycle**
  - Simple: DB session is cleaned after each request.
  - Technical: teardown performs rollback/remove to prevent leaked transactions/connections.

- **Health endpoint**
  - Simple: quick endpoint to check if app is alive.
  - Technical: `/health` returns static JSON status, suitable for probes but not deep dependency health.

- **CLI command integration**
  - Simple: backend exposes maintenance command.
  - Technical: custom Flask CLI command (`weekly-hard-cleanup`) executes domain cleanup service.

- **Audit logging**
  - Simple: important actions are recorded.
  - Technical: event logging service tracks module/action/entity/metadata for traceability.

- **Domain workflows**
  - Simple: defined business states for reservations, tickets, debts.
  - Technical: status constants and transition logic implemented across services/controllers.

- **Legacy alias normalization**
  - Simple: old role names are mapped to current names.
  - Technical: role normalization maps aliases (`USER`, `PROFESOR`, etc.) to canonical RBAC roles.

### B. Database design

- **SQLAlchemy ORM models**
  - Simple: Python classes represent DB tables.
  - Technical: declarative models with typed columns, constraints, relationships, and backrefs.

- **PostgreSQL connection URI**
  - Simple: app reads DB credentials from env vars.
  - Technical: connection string assembled from `DB_USER/DB_PASSWORD/DB_HOST/DB_PORT/DB_NAME`.

- **Primary keys / Foreign keys**
  - Simple: each row has ID and links to related rows.
  - Technical: integer PKs and FK references across user/material/reservation/ticket/debt/notification entities.

- **Unique constraints**
  - Simple: prevents duplicate combinations.
  - Technical: e.g., unique `(role, permission_id)` and `(ticket_id, material_id)` and `(user_id, endpoint)`.

- **Indexes**
  - Simple: speed up frequent searches.
  - Technical: explicit indexing on lookup/status columns (`event_code`, `ticket_id`, `request_date`, `status`, etc.).

- **Server defaults (`db.func.now`)**
  - Simple: timestamps auto-generated by DB.
  - Technical: `created_at` with server-side now for consistency regardless of app clock drift.

- **Many-to-one relationships**
  - Simple: many records can belong to one parent.
  - Technical: examples: reservationâ†’user, debtâ†’user/material/ticket, notificationâ†’user.

- **Association table for permissions**
  - Simple: connects roles to permissions.
  - Technical: `role_permissions` materializes RBAC grant mappings.

- **Soft state flags**
  - Simple: rows often marked by status instead of deleted immediately.
  - Technical: status-driven lifecycle (`PENDING`, `APPROVED`, `OPEN`, etc.) in operational entities.

- **Alembic migrations**
  - Simple: controlled DB schema versioning.
  - Technical: Flask-Migrate/Alembic online/offline migration environment configured from app engine.

### C. Security

- **RBAC (Role-Based Access Control)**
  - Simple: each role has different system permissions.
  - Technical: hierarchical role checks (`min_role_required`) and dynamic permission checks (`permission_required` + `role_permissions`).

- **Session authentication**
  - Simple: login state maintained with cookies.
  - Technical: Flask-Login `current_user`, user loader via DB session.

- **Password hashing**
  - Simple: password is not saved in plain text.
  - Technical: Werkzeug `generate_password_hash` and `check_password_hash`.

- **CSRF protection**
  - Simple: form/action tokens prevent forged requests.
  - Technical: per-session `_csrf_token`, enforced for mutating non-API routes, token accepted via form field or `X-CSRFToken`.

- **Content Security Policy (CSP)**
  - Simple: browser is told what external content is allowed.
  - Technical: `Content-Security-Policy` header set from explicit directive map (script/style/font/img/connect/frame/media sources).

- **API key validation**
  - Simple: API can require a shared key.
  - Technical: `X-API-Key` compared with `hmac.compare_digest`; allows either authenticated user or valid key.

- **In-memory API rate limiting**
  - Simple: too many requests are temporarily blocked.
  - Technical: per actor/source sliding window using deque + lock; configurable minute/window limits.

- **Secure cookie flags**
  - Simple: safer session cookie behavior.
  - Technical: `HTTPOnly`, `SameSite=Lax`, `Secure` enabled outside dev-like environments.

- **Environment hardening gate**
  - Simple: production refuses insecure default secrets.
  - Technical: app raises `RuntimeError` in non-dev env if key/salt/API key remain default.

- **Path traversal defenses for uploads**
  - Simple: blocks dangerous file paths.
  - Technical: normalizes path, rejects absolute/`..`, resolves candidate under allowed roots.

- **Account status gates**
  - Simple: blocked, inactive, unverified users are restricted.
  - Technical: RA and auth flows validate `is_active`, `is_banned`, `is_verified`, debt-block state.

### D. Performance

- **Header notification cache (TTL)**
  - Simple: avoids querying DB repeatedly in same time window.
  - Technical: in-process cache per user with lock and short TTL (5s) for header payload.

- **Query limiting / ordering**
  - Simple: returns only recent or bounded results.
  - Technical: frequent use of `order_by(...desc())`, `limit(5/200)`, and selective filters.

- **Joined eager loading (`joinedload`)**
  - Simple: fetch related data in one go.
  - Technical: used in schedule/professor assignment queries to reduce lazy-load overhead.

- **Bulk update/delete**
  - Simple: multiple rows updated quickly.
  - Technical: `update(..., synchronize_session=False)` and bulk delete patterns for notifications.

- **SSE heartbeat strategy**
  - Simple: periodic keepalive to keep connection healthy.
  - Technical: heartbeat every 8s to prevent stale sockets and gthread worker exhaustion.

- **Anti-duplicate notifications**
  - Simple: avoids sending same notification repeatedly.
  - Technical: short dedup time window query (`DEDUP_SECONDS_DEFAULT=3`).

- **Client-side fallback polling**
  - Simple: if realtime fails, app checks periodically.
  - Technical: JS fallback timer polls feed every 12s after SSE retries fail.

- **Scoped partial view refresh**
  - Simple: refreshes only changed section, not whole page.
  - Technical: fragment selector mapping + fetch + DOM replacement triggered on notification events.

### E. Frontend

- **Server-side rendered templates (Jinja)**
  - Simple: HTML is built on the server.
  - Technical: template inheritance/components with server-provided state and filters.

- **Progressive enhancement scripts**
  - Simple: JS adds dynamic behavior to rendered pages.
  - Technical: client scripts handle notification center, push lifecycle, AJAX actions, UI toasts.

- **Fetch API / AJAX**
  - Simple: browser sends background HTTP requests.
  - Technical: `fetch` for feed, mark-read, clear-read, push subscribe/unsubscribe, RA client actions.

- **Event-driven UI**
  - Simple: UI reacts to events like clicks/messages.
  - Technical: custom events (`coyolabs:notification_created`), visibility/page lifecycle listeners, service worker messages.

- **Client-side QR flow**
  - Simple: camera scans QR, then app validates and records event.
  - Technical: Html5Qrcode integration plus backend GET/POST RA endpoints with parsing and access checks.

- **Accessibility semantics**
  - Simple: UI has accessibility roles/labels.
  - Technical: ARIA roles, labels, dialogs, button semantics across templates.

### F. Infrastructure / deployment

- **Gunicorn runtime assumption (single worker for SSE consistency)**
  - Simple: recommended to run one worker for reliable realtime behavior.
  - Technical: explicit note says SSE broker is process-local; multi-worker can miss cross-worker events.

- **Environment-variable based configuration**
  - Simple: behavior changes by server env vars.
  - Technical: config object centralizes secrets, DB URI, API keys, VAPID keys, cookie security.

- **Cron-based maintenance integration**
  - Simple: scheduled shell script runs cleanup weekly.
  - Technical: bash script activates venv, sets `FLASK_APP`, executes cleanup CLI; includes cron example pathing.

- **WSGI app entrypoint**
  - Simple: startup script for server process.
  - Technical: `run.py` loads app factory and can run Flask dev server / gunicorn module import.

- **Database migration pipeline**
  - Simple: schema evolves via migration scripts.
  - Technical: Alembic env binds app engine and suppresses empty autogenerations.

### G. Real-time systems

- **Server-Sent Events (SSE)**
  - Simple: server pushes updates to browser over one long connection.
  - Technical: `/notifications/stream` yields event frames with `event:` and `data:` format and `text/event-stream` MIME.

- **In-process pub/sub broker**
  - Simple: memory queue forwards events to connected clients.
  - Technical: thread-safe broker maps `user_id -> client_id -> queue.Queue(maxsize=100)`.

- **Client identity for stream deduplication**
  - Simple: browser stores an ID so old stream can be replaced.
  - Technical: sessionStorage client UUID passed in stream query to enforce one active queue per client ID.

- **Heartbeat events**
  - Simple: periodic ping confirms channel still alive.
  - Technical: emits `heartbeat` event with UTC timestamp payload on queue timeout.

- **Reconnect/backoff behavior**
  - Simple: browser reconnects after stream errors.
  - Technical: 2-second reconnect timer, retry counter, polling fallback after threshold.

- **Proxy buffering control header**
  - Simple: asks reverse proxy not to buffer stream.
  - Technical: `X-Accel-Buffering: no` + `Cache-Control: no-transform` for low-latency delivery.

### H. PWA / Web Push

- **Service Worker**
  - Simple: background script for push handling.
  - Technical: `/sw.js` served from Flask route, handles `push`, `notificationclick`, `pushsubscriptionchange`.

- **Web Push protocol**
  - Simple: server can send browser notifications when app is closed.
  - Technical: pywebpush dispatch to stored subscription endpoints using VAPID credentials.

- **VAPID key validation**
  - Simple: checks push key format before use.
  - Technical: base64url decode check for uncompressed P-256 public key (65 bytes, prefix `0x04`).

- **Push subscription persistence**
  - Simple: each browser subscription is saved in DB.
  - Technical: `push_subscriptions` table stores endpoint/keys/user-agent/activity flag and unique user+endpoint.

- **Permission-state-aware UX**
  - Simple: button state changes depending on browser notification permission.
  - Technical: front-end handles `granted/denied/default`, syncs server subscription, and supports unsubscribe.

- **Offline-like resilience for notifications**
  - Simple: if user is already on site, avoids duplicate system notification.
  - Technical: service worker checks visible clients before showing push notification.

## 2) Presentation pressure questions and answers

### A) LOW technical knowledge questions

- **Q1. Why did you use roles like STUDENT and ADMIN?**
  - Strong answer: To ensure each person only sees and does what corresponds to their responsibility; this reduces accidental misuse and improves accountability.
  - Weak point: Some permissions depend on role hierarchy and others on explicit permission mapping, which can be harder to explain quickly.

- **Q2. What does "real-time notifications" mean here?**
  - Strong answer: The browser keeps a live connection so new alerts appear instantly without reloading the page.
  - Weak point: In this implementation, that instant delivery is tied to a single process memory broker.

- **Q3. If internet is unstable, does the app stop notifying?**
  - Strong answer: It first retries realtime connection and then switches to periodic polling to keep notifications updated.
  - Weak point: Polling is less immediate and can add extra load.

- **Q4. Why are there CSRF tokens in forms?**
  - Strong answer: They prove the request comes from the real user session and not from a malicious page.
  - Weak point: API-prefixed endpoints are exempt from this CSRF check by design.

- **Q5. What is the purpose of the service worker?**
  - Strong answer: It enables browser push notifications and handles notification click behavior even when page is not active.
  - Weak point: It is focused on push; it is not implementing full offline app caching.

- **Q6. Why use PostgreSQL instead of plain files?**
  - Strong answer: It supports reliable concurrent data operations, relationships, constraints, and transactional integrity for multi-user operations.
  - Weak point: Requires proper DB ops/monitoring and schema migrations discipline.

- **Q7. How do you prevent duplicate records in key places?**
  - Strong answer: Through database unique constraints and controlled update logic.
  - Weak point: Not every domain has strict uniqueness beyond IDs and selected composites.

- **Q8. Can someone access another user's notifications?**
  - Strong answer: Notification queries are always filtered by authenticated user ID and protected by role-guarded routes.
  - Weak point: Reliance on application-level filtering means test coverage is essential for regressions.

### B) HIGH technical knowledge questions

- **Q1. Why SSE and not WebSockets?**
  - Strong answer: Current requirement is serverâ†’client notifications; SSE is simpler over HTTP, lightweight, and natively reconnectable for this one-way stream pattern.
  - Weak point: Bi-directional realtime cases would require additional channels or migration.

- **Q2. How do you handle SSE in multi-worker Gunicorn?**
  - Strong answer: The code explicitly documents broker locality and warns that cross-worker delivery is not guaranteed.
  - Weak point: Process-local broker means horizontal scaling introduces notification consistency gaps.

- **Q3. What is your backpressure strategy for slow SSE consumers?**
  - Strong answer: Each subscriber queue has bounded size (`maxsize=100`); publish drops events when queue is full rather than blocking server threads.
  - Weak point: Dropped events are not replayed from persistent stream storage.

- **Q4. Is CSRF globally enforced?**
  - Strong answer: It is enforced for mutating methods, except endpoints under `api.` namespace where alternate auth (session/API key) applies.
  - Weak point: API endpoints using session cookies without explicit CSRF can be scrutinized depending on threat model.

- **Q5. How is API key compared safely?**
  - Strong answer: Uses constant-time `hmac.compare_digest` to reduce timing attack leakage.
  - Weak point: Shared API key model is coarse-grained compared to scoped tokens.

- **Q6. Why keep a role hierarchy and also permission table?**
  - Strong answer: Hierarchy simplifies broad access gates, while role-permission mapping supports fine-grained checks where needed.
  - Weak point: Dual authorization paths can drift if governance/testing is weak.

- **Q7. What protects file serving under `/uploads/<path>`?**
  - Strong answer: It normalizes path input, rejects traversal indicators, resolves against allowed roots, and checks file existence before serving.
  - Weak point: Multiple allowed roots increase operational complexity and audit surface.

- **Q8. Do you guarantee idempotency in notification publishing?**
  - Strong answer: There is short-window dedupe for recently identical notifications and client-side upsert by notification ID.
  - Weak point: No distributed dedupe key across workers/services.

- **Q9. How is push subscription integrity handled?**
  - Strong answer: endpoint+p256dh+auth are validated for presence, tied to user, and stale subscriptions are deactivated on 404/410 push responses.
  - Weak point: Token rotation and endpoint churn can still create temporary mismatches.

- **Q10. How do you control DB session state during long-lived SSE responses?**
  - Strong answer: The stream endpoint explicitly removes scoped DB session before entering loop and again during teardown/finally.
  - Weak point: Long-lived request concurrency still depends on WSGI worker/thread tuning.

- **Q11. Is rate limiting distributed?**
  - Strong answer: Current limiter is in-memory per process with lock and sliding window behavior.
  - Weak point: Limits are not globally shared across instances/processes.

- **Q12. How do you reason about consistency between push and SSE channels?**
  - Strong answer: Notification creation triggers SSE publish and push dispatch from shared service path.
  - Weak point: Partial failures can cause one channel to deliver while the other fails.

## 3) Potential weaknesses detected (as-is)

### Scalability

- Process-local SSE broker limits reliable fanout across multiple workers/processes.
- In-memory rate limiting and notification cache are per-process, not global.
- Long-lived SSE connections can pressure thread-based workers under peak concurrency.

### Security

- API authentication model allows either session or shared API key; key scope/rotation granularity is limited.
- CSRF enforcement excludes API namespace, which requires careful endpoint-by-endpoint threat modeling.
- CSP contains `'unsafe-inline'` for scripts/styles, which weakens strict XSS mitigation posture.

### Performance

- Notification unread count uses fresh DB count queries; heavy notification volume could amplify read pressure.
- Frequent partial-refresh and fallback polling may increase request volume during SSE instability.
- Some access-control checks involve repeated ORM queries (permission lookup per check path).

### Architecture decisions

- Mixed data-access style (direct ORM queries + service + repository) can blur boundaries in large teams.
- Authorization is split between role hierarchy decorators and permission-table checks.
- Real-time reliability depends on runtime topology assumptions (single-worker note in `run.py`).

---

## 4) High-pressure defense lines (quick speaking points)

- The system is **modular at route level** (blueprints) and **workflow-centric at service level**.
- Core operational entities are **transactional and relational** (reservations/tickets/debts/notifications).
- Security stack is **layered**: session auth, role gating, optional permission matrix, CSRF, CSP, API key checks, and secure cookie config.
- Realtime strategy is **SSE-first with graceful fallback polling**, and push notifications are integrated via service worker + VAPID.
- Main trade-off: **current realtime and rate-limit mechanisms are process-local**, favoring simplicity over distributed consistency.
